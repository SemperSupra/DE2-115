# DE2-115 SignalTap Implementation Plan

This document lists concrete SignalTap captures for the current bring-up blockers.

## 1. USB HPI Bridge (CY7C67200)

**Context:** The HPI bridge can drive write data, but CY7C67200 readback remains `0x0000`. Firmware has already verified the HPI register order and local Wishbone decode. Command-line `HPI0` source/probe captures show valid read cycles with reset released, and an Etherbone reset/sample-offset sweep from 0 to 60 cycles still returned only zeroes. The next debug step is external pin capture or a same-board comparison with a known-good Terasic USB bitstream.

Target clock domain:

- `sys` at 50 MHz for internal bridge/state signals.
- If using an external analyzer, sample fast enough to see 50 MHz-cycle-level HPI control timing.

External pads to tap:

- `usb_otg_data[15:0]`
- `usb_otg_addr[1:0]`
- `usb_otg_cs_n`
- `usb_otg_rd_n`
- `usb_otg_wr_n`
- `usb_otg_rst_n`
- `usb_otg_dreq`
- `usb_otg_int0`
- `usb_otg_int1`

Internal signals to tap:

- `cy7c67200_wb_bridge.state`
- `cy7c67200_wb_bridge.count`
- `cy7c67200_wb_bridge.latched_we`
- `cy7c67200_wb_bridge.write_data`
- `cy7c67200_wb_bridge.read_data`
- `cy7c67200_wb_bridge.last_sample_data`
- `cy7c67200_wb_bridge.last_cy_data`
- Wishbone `cyc`, `stb`, `we`, `ack`, `adr`, `dat_w`, `dat_r`

Triggers:

- Read capture: falling edge of `usb_otg_cs_n` when `usb_otg_rd_n == 0`.
- Write capture: falling edge of `usb_otg_cs_n` when `usb_otg_wr_n == 0`.
- Optional reset capture: rising edge of `usb_otg_rst_n`.

Sample depth:

- 2048 minimum.
- 4096 preferred if adding reset plus first register probes.

Pass/fail checks:

- During writes, FPGA should drive `OTG_DATA`.
- During reads, FPGA should tri-state `OTG_DATA` and CY should drive it after `CS_N/RD_N` assert.
- `OTG_RST_N` should be low until firmware releases reset, then remain high.

## 2. Ethernet RGMII (Marvell 88E1111, Port 1)

**Context:** PHY17 responds over MDIO and the current forced 10/100 image works. Host ping to `192.168.178.50` passed, and Etherbone CSR access through `litex_server` on TCP port `1235` read identifier memory and wrote/read LED CSRs. Gigabit RGMII cleanup is deferred.

Target clock domain:

- `eth_rx`/`eth_tx` as generated by the PHY wrapper for RGMII/LiteEth captures.

Signals to tap:

- LiteEth MAC sink/source valid/ready/data/last.
- `rgmii_eth` Port 1 RX/TX pads:
  - `rx_ctl`
  - `rx_data[3:0]`
  - `tx_ctl`
  - `tx_data[3:0]`
  - `rst_n`
- Optional: MDIO/MDC if changing PHY registers.

Triggers for later gigabit or robustness work:

- MAC RX valid assertion.
- RGMII `rx_ctl` assertion.
- TX valid assertion when probing ARP replies.

Sample depth:

- 4096 minimum.

Pass/fail checks:

- Forced 10/100 baseline should continue to pass ping.
- Etherbone CSR access should continue to work through a non-conflicting host TCP bind port.
- When revisiting gigabit, ARP request from host should appear on RX and ARP/UDP reply should leave TX.

## 3. VGA Output

VGA is currently not blocking Ethernet or USB work.

Only capture if visual regressions appear:

- `vga_r[7:0]`
- `vga_g[7:0]`
- `vga_b[7:0]`
- `vga_blank_n`
- `ping_hsync`
- `ping_vsync`
- internal `h_cnt` and `v_cnt`

## 4. Other Peripherals

LCD, 7-segment displays, LEDs, and SDRAM are baseline devices. The next substantive target after Ethernet/USB is SD card because LiteSDCard is already instantiated and can be tested with deterministic block and filesystem checks.
